# 協作情境練習

這份文件提供 3 個由淺入深的練習情境。建議照順序做。
每個情境都先說明「會發生什麼」，再給操作步驟。
基本操作（clone / branch / commit / push / PR）請先看 [教學指南.md](教學指南.md)。

---

## 情境 1：基本流程（無衝突）

**目標**：完整走一遍 branch → commit → push → PR → review → merge，且不會遇到衝突。

**設計**：每個人填 `members.md` 裡**不同**的成員區塊，互不干擾，所以不會衝突。

**步驟**
1. 認領一個還沒人填的「成員 N」區塊（先跟組員講好誰填哪一個）。
2. 建立 branch `feature/你的名字`。
3. 編輯 `members.md`，只改你那一個區塊。
4. commit：`feat: add 你的名字`。
5. push，開 PR 到 `main`。
6. 請一位組員 Approve，然後 merge。

✅ 學到：完整的 PR 協作流程、為什麼「各做各的區塊」能避免衝突。

## 情境 2：製造並解決 merge 衝突

**目標**：體驗兩個人改到「同一行」時的衝突，並學會解決。

**設計**：`衝突練習區.md` 裡有一行「🏆 本週最佳組員（MVP）」，大家都要改它，所以一定會衝突。

我已經先準備好一個**現成的衝突 PR**：branch `scenario/conflict-demo`（把 MVP 填成 Bob），
而 `main` 上同一行已被改成 Alice。打開那個 PR 就會看到 GitHub 顯示 **conflict**。

### 玩法 A：直接看現成的衝突 PR（適合一起示範）
1. 到 GitHub 的 **Pull requests**，打開來自 `scenario/conflict-demo` 的 PR。
2. 會看到 "This branch has conflicts that must be resolved"。
3. 按 **Resolve conflicts**，刪掉 `<<<<<<<`, `=======`, `>>>>>>>` 標記，決定最後要留誰，存檔 → **Mark as resolved** → **Commit merge**。

### 玩法 B：自己製造一個（兩人一組）
1. 兩個人**各自**開 branch，都去改 `衝突練習區.md` 的同一行 MVP。
2. 兩人都 push、開 PR。
3. **第一個** merge 的人會順利；**第二個**就會遇到衝突。
4. 第二個人在自己 branch 上解決衝突（見下方），再 merge。

### 解衝突時你會看到
```
<<<<<<< HEAD（main 上的版本）
🏆 本週最佳組員（MVP）：Alice
=======
🏆 本週最佳組員（MVP）：Bob
>>>>>>> scenario/conflict-demo（你的版本）
```
把三行標記（`<<<`, `===`, `>>>`）和你不要的那行刪掉，只留最終結果，例如：
```
🏆 本週最佳組員（MVP）：Alice
```

- **GitHub Desktop / 網頁**：用上面「玩法 A」的按鈕流程。
- **Claude Code**：對 Claude 說「幫我解決 `衝突練習區.md` 的衝突，最後留 Alice」。

✅ 學到：衝突為什麼發生、衝突標記怎麼讀、怎麼安全地解掉。

## 情境 3：branch 落後 main，先同步再 merge

**目標**：當你的 branch 開出來之後 main 又被別人更新了，學會「先把 main 的更新拉進來」再 merge。

**設計**：我已準備 branch `scenario/behind-main`，它有自己的修改（填了某個成員區塊），
但 `main` 在它之後又多了一個 commit，所以這個 branch **落後 main 一個 commit**（兩者沒有改到同一處，所以可以乾淨同步、不會衝突）。

**步驟**
1. 到 GitHub 打開來自 `scenario/behind-main` 的 PR（或自己開）。
2. 你會看到 "This branch is out-of-date with the base branch"。
3. 按 **Update branch**（GitHub Desktop 則用 **Branch → Update from main**）。
   - Claude Code：說「幫我把 main 的更新合併進這個 branch」（相當於 `git merge main` 或 `git pull origin main`）。
4. 同步後 PR 變成可 merge，請組員 Approve 後 merge。

> 小知識：如果同步時剛好改到同一處，就會變成情境 2 的衝突，處理方式一樣。

✅ 學到：branch 落後 main 是常態，merge 前要先同步；以及 Update branch 按鈕的用途。

---

## 建議練習順序

1. 全員先做**情境 1**，熟悉流程。
2. 再做**情境 2**，體驗衝突（先看玩法 A 示範，再玩法 B 自己做一次）。
3. 最後**情境 3**，理解 branch 同步。
